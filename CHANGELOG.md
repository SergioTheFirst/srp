## [Unreleased]

### Added

- Развёртывание агента как службы под LocalSystem: скрипты
  `client/deploy/install-service.ps1` / `uninstall-service.ps1` (задача
  Планировщика — запуск при старте от SYSTEM, рестарт при сбое; пишет
  `server_url` в конфиг и проверяет одним проходом перед установкой). Запуск под
  SYSTEM разблокирует привилегированные сборщики (SMART/WMI). Раздел
  «Развёртывание» в README, включая заметку про TLS на reverse-proxy.
- Лог-файл агента для службы: `--log-file PATH` (ротация ~1 МБ × 3); стартовая
  строка лога пишет `user=…`, чтобы оператор подтвердил запуск от SYSTEM.
- Агент собирает метаданные установленных личных сертификатов (субъект, издатель, отпечаток, срок действия) из хранилищ LocalMachine\My и CurrentUser\My; дашборд устройства подсвечивает истекающие и просроченные сертификаты.
- API флота `/api/v1/devices` обогащён для живого дашборда: доверие, число
  UNKNOWN-доменов, пропавшие источники, staleness (агент молчит), ближайший срок
  сертификата; добавлено квитирование устройства оператором (`POST /devices/{id}/ack`
  с заметкой) — канал обратной связи.
- Живой дашборд флота: автообновление (~12с, swap фрагмента), группировка по
  объекту, KPI-фильтры (риск / UNKNOWN / пропавшие источники / не на связи /
  сертификаты / untrusted), поиск и квитирование устройств прямо из списка.

### Changed

- История здоровья теперь накапливается, а не перезаписывается: показания
  `historical` и вычисленные оценки (`scores`) хранятся append-only (по записи на
  сбор), что делает возможными тренды («становится ли хуже?») и будущую привязку
  меток. Дашборд по-прежнему показывает последнее значение. Существующие БД
  мигрируются автоматически при старте сервера, без потери данных (P0/W0.1).
- Серверное время (P0/W0.2): сервер штампует `received_at` (момент приёма) на
  телеметрии; staleness («не на связи»), окна событий и тренды считаются по нему,
  а не по часам клиента (часы ПК могут врать — событие/сэмпл берёт системное
  время машины). `ts` сохранён как время клиента (совместимость). Расхождение
  `received_at`−`ts` выводится как сигнал `clock_drift` (порча трендов и/или
  старение CMOS-батарейки). Существующие БД мигрируются автоматически.

### Security

- Удалён небезопасный дефолт адреса сервера: `server_url` больше не указывает по
  умолчанию на публичный IP. Оператор задаёт адрес при установке (`client/config.json`
  или `--server`) — обычно сервер в локальной сети, публичный адрес возможен явно.
  Пустой `server_url` = агент не стартует (ошибка с подсказкой), а не молчит в неизвестный хост.
- Аутентификация ingest: опциональный общий токен (`ingest_token`). Если задан —
  `/api/v1/ingest` требует заголовок `X-SRP-Token` (иначе 401); агент шлёт его.
  Пусто = выключено (MVP-совместимо). Закрывает открытый ingest (спуфинг /
  отравление телеметрии). TLS — на reverse-proxy (вне кода).
- `CLAUDE.md` — гид для разработки: команды, жёсткие инварианты, обязательное
  применение skills и субагентов.
- Хук `.claude/hooks/ruff_format.py` (PostToolUse) — авто `ruff check --fix` +
  `ruff format` для изменённых `.py`.
- Память проекта `.claude/memory/` — индекс `MEMORY.md` и заметки об инвариантах.
- Этот `CHANGELOG.md`.
- Слой доверия телеметрии (`server/trust/`): классификация источников
  `state` (гейт) + `weight` (модуляция), collector⊥semantic, объяснимое
  разрешение доверия по доменам (контракт — `telemetry-trust-contract.md`).
- Агент сообщает здоровье каждого источника телеметрии (`source_health`) в каждом
  конверте — статус сбора (`ok/partial/empty/timeout/blocked/absent`) по каждому
  источнику; сервер сможет отличать «источник недоступен/пуст» от «здоров»
  (недоступность ≠ здоровье).
- Сервер вычисляет доверие по источникам/доменам на каждом приёме (lineage) и
  размечает им риск: класс отказа с недостоверным доменом показывается как
  UNKNOWN (без ложно-уверенной вероятности); устройство с недостоверной
  идентичностью помечается `untrusted`, а его day-1 оценки не показываются (§7).
- Обнаружение «пропавших» источников: источник, который раньше отдавал данные, а
  теперь не отвечает (`empty/timeout/blocked/absent`), помечается `regressed` и
  выводится баннером на дашборде — отличается от никогда не виденного источника.
- Привязка устройства к объекту/организации: вручную присваиваемый `site_code`
  (+ опц. `site_name`) приходит в конверте и хранится на устройстве; флот
  группируется по объекту (масштабирование на множество фирм/площадок).

## [0.1.0] - 2026-05-29

Первый MVP системы раннего предупреждения отказов.

### Added

- Тонкий агент (`client/`) на чистом стандартном Python: сбор `inventory`,
  `historical`, `heartbeat`, `events` через PowerShell; отправка в общем
  конверте с офлайн-буфером (`buffer.jsonl`, FIFO-досылка).
- Сервер (`server/`) на FastAPI + SQLite: приём `/api/v1/ingest`, хранение
  истории, пересчёт оценок на каждое сообщение.
- 4 оценки «день-1» (`performance`, `reliability`, `wear`, `risk_exposure`) и
  байесовский риск по классам отказа в лог-оддсах с объяснимым вкладом признаков.
- Веб-дашборд парка и устройства (серверный рендер Jinja2, autoescape вкл.).
- Контракт сообщений — единый источник истины (`shared/schema.py`, pydantic v2).
- Качество кода: `ruff`, `mypy`, `bandit`, `pytest` (порог покрытия 80 %),
  `pre-commit`, GitHub Actions CI, `Makefile`, `smoke.py`.

### Security

- Серийные номера дисков хешируются (SHA-256) в агенте; сырой серийник не
  покидает процесс.
- Jinja2 autoescape включён; весь SQL параметризован; вход валидируется на
  границе через pydantic.
