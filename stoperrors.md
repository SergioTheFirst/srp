# stoperrors.md — план исправления найденных ошибок

> Сгенерировано автоматическим код-ревью (`/code-review`, effort=high, 12 параллельных
> read-only агентов по подсистемам + 2 адверсариальных верификации трудных находок).
> Диапазон: весь репозиторий (текущая ветка идентична `main` — рабочего diff не было,
> поэтому ревью прошло по всей кодовой базе, а не по патчу).
> Дата ревью: 2026-07-18. Исполнитель плана: **Claude Code (Sonnet)**, по одному
> атомарному пункту за раз. **Код в этом файле не пишется** — только план.

## 0. Как пользоваться этим планом (обязательно к прочтению перед стартом)

1. **Один пункт = одна ветка = один коммит = один прогон гейта.** Не объединять
   несвязанные пункты в один коммит, даже «мелкие». Причина: если гейт упадёт после
   объединённого коммита, откат/бисект становится дороже — а часть находок ниже сама
   является примером того, к чему приводит «на скорую руку» правка соседнего кода.
2. **TDD RED→GREEN на каждый пункт**: сначала тест, воспроизводящий сценарий отказа
   из описания находки (падает → RED), затем минимальная правка (тест зеленеет →
   GREEN). Тест остаётся в репозитории — это и есть защита от регресса находки.
3. Пока работаете над пунктом — гоняйте **только** релевantные тестовые файлы
   (`pytest tests/test_x.py -q`). Полный гейт (`ruff` · `mypy` · `bandit` · `pytest
   cov≥80%` · `python smoke.py`) — **один раз**, непосредственно перед `merge --no-ff`
   этого пункта, не по ходу итерации.
4. **Обязательное ревью субагентом перед merge**:
   - `security-reviewer` (или Opus-класс модели в роли ревьюера, если отдельного
     агента `security-reviewer` нет в реестре) — ОБЯЗАТЕЛЕН для любого пункта,
     трогающего: `client/` (агент), ingest/auth-поверхность (`server/api.py`,
     `server/main.py`, `server/ingest_guards.py`), SQL (`server/db.py`),
     `server/trust/`, `server/scoring/bayesian.py`, сертификаты/приватность
     (`client/collectors/user_certs.py`, `smart.py`, `network.py`), обновление
     агента (`client/updater.py`, `server/updates.py`), установку
     (`client/deploy/setup.py`).
   - `code-reviewer` (Sonnet) — для всех остальных пунктов (analytics, netdisco,
     printers-логика без BER-парсера, dashboard, scores).
5. **Пункты, помеченные ⚠️ ТРЕБУЕТ РЕШЕНИЯ ПОЛЬЗОВАТЕЛЯ**, нельзя реализовывать
   автономно без явного подтверждения — они меняют поведение по умолчанию для уже
   развёрнутого флота или удаляют существующее (пусть и мёртвое) поведение. Если вы
   Claude Code, выполняющий этот план — **остановитесь и спросите** перед этими
   пунктами, не додумывайте намерение.
6. Каждый пункт с видимым для оператора эффектом → строка в `CHANGELOG.md`
   (`## [Unreleased]`) в том же коммите. Обновить `CONTINUITY.md` после каждого
   пункта (что сделано, что дальше).
7. Номера строк ниже — **ориентир на момент ревью**; строки могли сместиться из-за
   предыдущих пунктов плана. Перед правкой найдите символ через `.codegraph/`
   (или grep, если codegraph устарел) и подтвердите текущий номер строки.
8. Никаких `git add -A`. Стейджить только реально изменённые файлы данного пункта.
9. Порядок фаз обязателен: **P0 → P1 → P2 → P3**. Внутри фазы пункты независимы
   (кроме случаев, где явно указана зависимость) — можно в любом порядке, но не
   параллельно одним агентом (файлы могут пересекаться, см. зависимости).

Легенда серьёзности: 🔴 P0 критично · 🟠 P1 высокая · 🟡 P2 средняя · ⚪ P3 низкая/уборка.
Модель/effort — по таблице CLAUDE.md §2 (напоминание в каждом пункте).

### P0-1 🔴 Неаутентифицированные деструктивные эндпойнты `delete_device` / `purge_devices`

### P0-2 🔴 ⚠️ ТРЕБУЕТ РЕШЕНИЯ ПОЛЬЗОВАТЕЛЯ — ingest-аутентификация выключена по умолчанию и не включена в поставке
- **Файлы**: `server/config.py:46` (`ingest_token: str = ""`), `server/main.py:416`,
  `server/config.json` (ключа `ingest_token` нет вообще).
- **Симптом**: правило CLAUDE.md «нет фич, выключенных по умолчанию — включи в
  `server/config.json` или сделай переключаемой из дашборда» соблюдено для
  `printer_poll_enabled`/`netdisco_enabled`, но не для единственного
  security-relevant дефолта — токена. Как поставляется — `/api/v1/ingest`
  принимает телеметрию от кого угодно в сети без токена.
- **Почему это ⚠️ решение, а не автономная правка**: если Sonnet молча сгенерирует
  случайный токен и впишет его в `server/config.json`, это **сломает приём
  телеметрии от уже развёрнутых агентов**, у которых `client/config.json` не содержит
  этот токен (шаблон `client/config.json` пуст и не коммитится — см. CLAUDE.md §6).
  Это прямое нарушение «не навреди» — тихая порча продакшен-флота.
- **Что сделать (после подтверждения пользователем одного из вариантов)**:
  - Вариант А (безопаснее, рекомендуется): при пустом `ingest_token` сервер при
    старте выводит явное **громкое предупреждение** в лог (уровень WARNING/ERROR)
    вида «ingest_token не задан — сервер принимает телеметрию БЕЗ аутентификации»,
    и то же предупреждение — баннером на дашборде (`server/web/dashboard.py`),
    пока токен не задан. Поведение приёма не меняется — только видимость проблемы.
  - Вариант Б (ломает обратную совместимость, только с явного согласия и плана
    миграции агентов): сгенерировать токен при первом запуске, записать в
    `server/config.json` и потребовать координированного обновления
    `client/config.json` на всём флоте одновременно.
  - **Не приступать к кодингу, пока пользователь не выберет вариант.**
- **Тест**: для варианта А — тест на наличие warning-лога/баннера при пустом токене
  на старте.
- **Ревью**: security-reviewer.
- **Модель**: Opus · max (R4 — меняет security-поверхность/контракт по умолчанию,
  требует явного решения — ambiguous ask по таблице §2).

### P0-3 🔴 Привилегированная эскалация из-за порядка ACL при установке агента
- **Файл**: `client/deploy/setup.py:398-403` (и повтор паттерна на `486-508` для
  апдейта); правильный порядок для сравнения — `spool`-директория на `406-411`.
- **Симптом**: `robocopy` копирует `srp-agent.exe`/`srp-tray.exe`/задачу планировщика
  в `C:\SRP`, ПОТОМ применяется `icacls /inheritance:r /grant:r` без `/T`. NTFS
  ACL наследуется в момент создания файла (copy-on-create), а не «живьём» —
  файлы, скопированные до `icacls`, остаются с унаследованным от `C:\`
  разрешением `Users:WRITE` навсегда.
- **Причина**: порядок операций перепутан; тот же разработчик сделал правильно для
  `spool` (создание после `icacls`), но не для основной полезной нагрузки.
- **Что сделать**:
  1. Поменять порядок: сначала создать целевую директорию (`destp.mkdir()`),
     сразу применить `icacls_cmd(dest)`, и только потом `robocopy`.
  2. Дополнительно (defense-in-depth, не замена шага 1) добавить `/T` к команде
     `icacls` в `icacls_cmd`, чтобы рекурсивно перевыставлять ACL на существующее
     дерево — на случай апдейта поверх старой установки, где шаг 1 уже неприменим
     ретроактивно.
  3. Применить тот же порядок к пути обновления (`run_update`, строки ~486-508).
- **Тест**: юнит-тест на порядок вызовов (mock `robocopy_cmd`/`icacls_cmd`,
  проверить порядок вызовов через `call_args_list`); интеграционный тест на
  Windows-раннере, если он есть в CI — проверить фактическую ACL после установки
  (если CI без Windows-раннера, ограничиться юнит-тестом порядка + ручной
  верификацией в описании PR).
- **Ревью**: security-reviewer (privilege escalation, установка агента).
- **Модель**: Opus · max (R4 — агент/приватность/установка, PowerShell-соседний
  код; после аппрува план-пункта конкретное исполнение может идти как R2, но
  ревью — обязательно Opus-класса ввиду privesc).

### P0-4 🔴 `ingest_token` передаётся в открытом виде и одновременно используется как HMAC-ключ обновлений
- **Файлы**: `client/updater.py:303-320,322-363` (использование токена как HMAC
  secret), `client/transport.py:164` (токен в заголовке `X-SRP-Token` на каждом
  обычном HTTP-запросе).
- **Симптом**: HMAC в `updater.py` существует specifically, чтобы agent мог
  проверить, что манифест/пакет обновления не подменён MITM-атакующим в той же
  локальной сети. Но ключ для этого HMAC — тот же `ingest_token`, что летает в
  открытом виде в заголовке каждого heartbeat/telemetry POST. Атакующий, пассивно
  слушающий один такой запрос, получает ключ и может подписать поддельный
  манифест/пакет, который агент примет и выполнит от имени SYSTEM.
- **Причина**: один и тот же секрет используется в двух ролях (bearer-токен
  аутентификации ingest И ключ подписи обновлений), что нарушает базовый принцип
  разделения ключей по назначению.
- **Что сделать**:
  1. Добавить отдельное поле секрета для подписи обновлений, например
     `update_hmac_secret: str = ""` в `server/config.py` (по аналогии с
     `ingest_token`) и соответствующее поле в `client/config.py`.
  2. `server/updates.py` при вычислении HMAC манифеста/пакета использовать
     `update_hmac_secret`, если задан, иначе fallback на `ingest_token` с явным
     WARNING-логом «update_hmac_secret не задан, используется ingest_token —
     менее безопасно, т.к. этот токен также передаётся как bearer-заголовок».
  3. `client/updater.py` аналогично — проверять HMAC новым секретом, если сервер
     его прислал/использует (нужно согласовать протокол обмена — секрет для
     проверки на стороне агента уже должен быть у агента локально в конфиге, как
     и `ingest_token` сейчас; сервер не должен пересылать сам секрет по сети).
  4. Не менять транспорт `ingest_token` в заголовках — это отдельная, более
     дорогая задача (mTLS/HTTPS-принуждение), вне рамок этого бага; текущий фикс
     устраняет именно повторное использование одного секрета в двух ролях.
- **Тест**: тест, что HMAC-проверка использует `update_hmac_secret`, когда он
  задан, и не совпадает с результатом при использовании одного `ingest_token`
  (т.е. компрометация ingest_token сама по себе больше не даёт валидный HMAC
  обновления). Существующие `test_update_status_ingest.py` не должны сломаться.
- **Ревью**: security-reviewer (обновление агента — supply chain).
- **Модель**: Opus · max (R4 — agent/ingest/обновление, security-контракт).

- **Причина**: `compute_risk`/хелперы не принимают домен-trust вообще как вход —

  1. Добавить параметр (например `domain_trust: dict[str, str]` — состояние

     класс-хелпер (`_storage`, `_power_thermal`, `_memory`, `_stability`).
  2. В каждом хелпере: если соответствующий домен НЕ в состоянии TRUSTED —
     возвращать `probability=None`/`level="unknown"` (а не считать число и

     реальный trust и убрать(или оставить как safety-net, но не единственный
     механизм) постфактумную аннотацию.
  4. Обновить `GET /api/v1/devices/{id}` — поле `probability`/`level` для
     untrusted-домена должно реально быть `null`/`"unknown"` в JSON-ответе, а не
     только скрываться в HTML.
- **Тест**: RED — тест дергает `compute_risk` с историческими данными, явно
  указывающими на критический риск, но с `domain_trust={"storage": "untrusted"}`

  тесты фиксировали именно баг, актуализировать тест, не откатывать фикс).
- **Ревью**: security-reviewer (`server/trust/` смежная логика, ядро доверия).
- **Модель**: Opus · max (R4 — `server/trust/`-смежная зона, scoring weights,

### P0-6 🔴 NULL-телеметрия BSOD/KP41 в когортах считается «здоровой» (0%) вместо исключения
- **Файл**: `server/db.py` — `get_fleet_cohort_stats`, строки ~3565-3568
  (`bsod_pct`, `kp41_pct` для когорты) и ~3596-3597 (`kp41_pct` для сайта).
  Правильный паттерн — тремя строками ниже в том же запросе, `rsi_low_pct`
  (~3569-3575), где стоит `... IS NOT NULL AND CAST(...) < 5.0` перед подсчётом.
- **Симптом**: `CAST(json_extract(...) AS REAL) >= 1` при NULL даёт NULL →
  `CASE` уходит в `ELSE 0.0` — устройство, которое просто не репортит это поле
  (старый агент), засчитывается как «BSOD не было», а не исключается из
  знаменателя. Когорта, где 90% устройств физически не могут прислать это поле,
  показывает искусственно низкий процент вместо честного «мало данных».
- **Причина**: `CASE WHEN ... THEN 1.0 ELSE 0.0 END` без явного `IS NOT NULL`
  guard — паттерн был явно известен автору (применён для `rsi_low_pct` в той же
  функции), но не скопирован на соседние два агрегата.
- **Что сделать**: добавить `AND json_extract(h.payload,'$.bugchecks_30d') IS NOT
  NULL` (аналогично для `kernel_power_41_30d` в обоих местах — cohort и site) в

### P0-7 🔴 Несовпадение формата временных меток ломает staleness-метрики и retention-обрезку
- **Файлы**:
  - `server/db.py:3939-3942,3959-3969` (`get_pipeline_metrics`: `stale`,
    `hb_5m`, `hb_1h`, `hist_5m`, `hist_1h`).
  - `server/db.py:1282-1301` (`prune_aged`) + `1203-1253`
    (`_archive_aged_rows`) — тот же корень.
- **Симптом**: Python хранит `received_at`/`last_seen` в ISO с разделителем `T`
  (`_now_iso()` → `"2026-07-18T01:35:00.413166+00:00"`), а SQL-фильтры сравнивают
  с `datetime('now', ...)`, которая возвращает строку с пробелом
  (`"2026-07-18 01:26:00"`). SQLite сравнивает TEXT лексикографически, `'T'`
  (0x54) > `' '` (0x20) — при совпадающей календарной дате сравнение почти
  всегда даёт неверный результат независимо от времени суток. Подтверждено
  эмпирически ревью-агентом (устройство, «молчащее» 60 минут при пороге 600с,
  НЕ попадает в `stale`; окно «последние 5 минут» считает весь день). То же для
  retention: строки на границе календарного дня не архивируются/не удаляются в
  срок — окно хранения сырых данных «плывёт» на сутки.
- **Причина**: две разные функции формирования временной метки (Python
  `_now_iso()` и SQLite `datetime()`) используются как взаимозаменяемые, хотя
  дают текстуально разные форматы.
- **Что сделать** (выбрать ЕДИНЫЙ формат и провести его всюду, не точечно):
  1. Определить единый канонiчный формат хранения `received_at`/`last_seen` —
     рекомендуется оставить существующий формат `_now_iso()` (уже везде
     сохранено в БД, миграция данных не нужна) и **строить cutoff-значения в
     Python** тем же форматтером (`_now_iso()`-совместимая функция с
     `timedelta`), а не через `datetime('now', '-N minutes')` в SQL-тексте.
  2. Заменить в `get_pipeline_metrics` каждое использование
     `datetime('now', '-N ...')` внутри SQL на связанный параметр `?`, значение
     которого — Python-строка в том же формате, что и `_now_iso()`.
  3. То же самое в `prune_aged`/`_archive_aged_rows` для `hb_cutoff`/`ev_cutoff`
     (сейчас строятся через `strftime("%Y-%m-%d %H:%M:%S")` — переписать на
     формат `_now_iso()`).
  4. НЕ трогать в этом пункте f-string-интерполяцию `days` (это отдельно, P2-15)
     — здесь фикс только формата даты/времени, без изменения способа передачи

### P0-8 🔴 `ber.py::decode_tlv` принимает усечённый/завышенный length без проверки границ буфера
