## 0. Ведущий тезис (циничный, без подыгрывания)

Главный риск проекта — **не «небезопасный агент»**, а построить предиктивную систему
на недостоверной телеметрии и потом месяцами крутить ML поверх ложных данных. Windows
как источник **врёт**: `MSAcpi_ThermalZoneTemperature` — мусор на половине OEM, SMART/NVMe
поля vendor-specific, WMI-провайдеры висят, Event ID плывут между билдами, коллектор без
прав SYSTEM возвращает пусто. Поэтому слой доверия телеметрии — **P0 без компромиссов,
раньше любой аналитики.**

Второй тезис: **предсказуемость доменно-ограничена.** Реально прогнозируемо — износ
накопителя, заполнение диска, тренд boot-time, fleet-anomaly, throttle-trend,
driver-regression. Остальное (random BSOD, VRM/PSU/мать, intermittent, док) — постфактум.
Строить «оракул отказов» поверх непрогнозируемого = дорогой noise amplifier.

Третий тезис: **в прогнозируемых деплеция-доменах ML не нужен.** SSD %used /
disk-fill — это детерминированная арифметика (порог + slope + ETA), а не survival и не
классификатор. ML/survival/петля меток — **отложенные гейтованные фазы**, строятся только
на доказанной необходимости, не как центр системы.

## 1. Решения и инварианты (decision record)

| # | Решение | Почему |
|---|---------|--------|
| D1 | **Telemetry-trust = P0, до аналитики.** | Garbage in → Bayesian garbage out. Фатальнее, чем отсутствие TLS-rotation. |
| D2 | **«Unknown != healthy» enforced end-to-end.** | Сейчас коллектор отдаёт `None`, но скор всё равно стартует со 100 → дохнущий диск без прав выглядит здоровым. |
| D3 | **Прогноз только в предсказуемых доменах.** Непрогнозируемое — инцидент/корреляция для показа, не prediction. | Не плодить false positives там, где сигнала нет. |
| D4 | **Wear-домены = тренд-математика + ETA, НЕ ML.** | survival там избыточен, а не «слабо калиброван». |
| D5 | **Тонкий объяснимый Bayesian-приоритизатор остаётся.** Никакого unified ML-оракула. | Объяснимость by-construction — конкурентное преимущество (`bayesian.py`). |
| D6 | **KP41 и WHEA — только correlation/burst-enhancer, не самостоятельные драйверы.** | KP41 specificity ≈ 0; WHEA в основном firmware/ASPM-шум. Сейчас в коде они прямые драйверы — дефект. |
| D7 | **Deployable architecture, но НЕ deployment-driven sequencing.** | Форсировать prod-hardening = installer/cert/ACL/GPO hell при всё ещё мусорных данных. |
| D8 | **Scope = Phased/gated.** survival + label loop + per-device anomaly — запланированы, но за гейтом доказанной нужды, не убиты. | Сохранить опцию, не строить собор ради ~20% непрогнозируемого остатка сейчас. |
| D9 | **Retain compressed raw windows для репроцессинга.** | Premature compression убивает ретроспективный feature-mining: «нашли новый precursor» → а данных нет. |

### Анти-цели (НЕ строить сейчас — premature infra gravity)
- Unified ML-оракул / «general PC health AI» / random-death prediction.
- RBAC-cathedral, PKI-empire, multi-tenant policy orchestration. (Числовые org/dept *ярлыки* — OK как расширение `site_code`, см. §3; *изоляция данных / RBAC / per-org секреты* — сюда, до доказанной нужды: реальный 2-й взаимно-недоверяющий клиент.)
- Auto-remediation, fleet-wide remote execution, сложный updater.
- Survival/labels как центр до прохождения гейтов §7.

### Сделано вне основной дорожной карты (отдельные RFC/планы)
- [x] **Мониторинг сетевых принтеров** (`server/printers/`): обнаружение (ARP-подсказки агентов + локальная печать ПК + активный RFC1918-скан за стоп-гейтом), SNMP-опрос (счётчики/расходники/ошибки, IPP/HTTP-резерв, вендор-драйверы), история, дашборд «принтеры» со сверкой ПК↔принтер.
- [x] **Network Discovery** (`server/netdisco/`, RFC `docs/superpowers/specs/2026-06-20-network-discovery-rfc.md` + план, **13/13 фаз готовы, в `origin/main`**): персистентный L2/L3 граф топологии (LLDP/CDP/FDB/ARP evidence-fusion, confidence-per-edge), пассивный SNMP-harvest, reachability-корреляция (DOWN vs UNREACHABLE), change-detection/ghost-lifecycle, read-only topology API + TTL-кэш + телеметрия, DPAPI-хранилище непубличной SNMP-community. Инварианты: OFF-by-default за двойным гейтом, RFC1918-only, SNMP read-only, контракт additive, агент не тронут. Память: `[[project-netdisco]]`.
- [x] **Объединение карты сети (Netmap-Unification)** (план `docs/superpowers/plans/2026-06-24-network-map-unification-plan.md`, **10/10 фаз готовы, в `origin/main`**): «Топология» и «Карта сети» слиты в ОДНУ «Карту сети». MAC-хребет идентичности (FK-связка net_devices↔agent/printer, Ф1) → единый read-side ассемблер (Ф2) → один API+GraphCache (Ф3) → один canvas-движок `_netgraph.html` (Ф4) → операторская панель управления: фильтры/слои/раскладки/боковая-панель/путь/первопричина/машина-времени/экспорт (Ф5) → одна каноническая карточка устройства (Ф6) → Tier-1 SNMP-углубление: LLDP-порты/VLAN/ifXTable/реальные wireless-рёбра/ENTITY-модель (Ф7) → Tier-2 пассивная де-анонимизация: reverse-DNS/mDNS/SSDP/NetBIOS/WSD/TLS-баннер (Ф8) → **Tier-3 опц-адаптеры** по кредам оператора: фреймворк + MikroTik/UniFi/Redfish/NetFlow + merge-by-MAC enrich-only + link-merge (Ф9 a–e) → демонтаж «Топологии», `/topology`→301 на `/netmap` (Ф10). Инварианты: идентичность по MAC (IP только явный резерв), адаптеры read-only/fail-soft/RFC1918/DPAPI-креды, enrich-only (никогда не перетирает SNMP/agent), агент не тронут. Память: `[[netmap-unification]]`, `[[netmap-adapters]]`.

  - **GATE:** калибровка (Brier/reliability) бьёт детерминированный ETA-baseline в shadow-режиме. Иначе не промоутить.

## 9.5 Бэклог из stoperrors.md код-ревью (2026-07-18), не входит в атомарные пункты плана
- **`get_fleet_cohort_stats`: AVG() по всей когорте → NULL, если НИ ОДНО устройство
  не репортит поле (найдено при реализации P0-6):** фикс P0-6 корректно исключает
  ОТДЕЛЬНОЕ устройство без поля из знаменателя `AVG()` per-метрика. Но если ВСЕ
  устройства когорты (не только часть) не репортят конкретное поле (например
  `bugchecks_30d` — старые агенты), `AVG()` по всей колонке возвращает SQL NULL →
  Python `float(row["bsod_pct"] or 0.0)` тихо подставляет 0.0 — тот же класс
  «UNKNOWN как healthy», только на уровне всей когорты, а не одного устройства.
  `compute_fleet_anomaly_risk` при этом видит только `cohort_size` (кол-во
  устройств), не «сколько из них реально репортят именно это поле» — confidence
  считается по размеру когорты, не по полноте поля. Не исправлено — требует
  либо доп. поля «cohort_reporting_N» на метрику, либо смены `or 0.0` на честный
  `None` с последующим аудитом `fleet_anomaly.py` (много существующих тестов
  предполагают float, не None) — за рамками точечного P0-6.
- **Owner-hijacking `C:\SRP` (найдено security-reviewer при верификации P0-3,
  `fix/p0-3-acl-order` → `main`):** P0-3 закрыл copy-on-create ACL-таймин (icacls
  до robocopy), но НЕ владельца директории. `C:\` по умолчанию доступен на запись
  обычным Users → атакующий может ЗАРАНЕЕ создать `C:\SRP` (или воткнуть файл в
  окне между `mkdir` и `icacls`) и стать его owner'ом. `icacls /grant:r` не меняет
  owner; владелец объекта NTFS ВСЕГДА имеет неявный WRITE_DAC независимо от DACL
  (фундаментальный инвариант NT ACL, не отключается без явного OWNER_RIGHTS/
  S-1-3-4 ACE) — то есть атакующий-владелец может в любой момент переоткрыть DACL
  и подменить `srp-agent.exe`, который выполняется как SYSTEM.
  Митигация (не реализована, нужна отдельная верификация): `icacls dest /setowner
  *S-1-5-32-544 /T` ПЕРЕД `/inheritance:r`/`/grant:r` — но `/setowner` в icacls.exe
  синтаксически ОТДЕЛЬНАЯ форма команды (не комбинируется с `/grant`/`/inheritance`
  в одном вызове, подтверждено `icacls /?` на живой машине) → нужен отдельный
  `_run` вызов + тест на порядок. Радикальная альтернатива: ставить в
  `%ProgramFiles%\SRP` (не user-writable) вместо `C:\SRP` — убирает вектор целиком,
  но это уже совместимость с существующим флотом (путь `C:\SRP` — часть контракта
  установки). Требует R4-решения (агент/установка/privesc), не автономный патч.
- **Несовпадение update_hmac_secret неотличимо от MITM (найдено security-reviewer
  при верификации P0-4, `fix/p0-4-update-hmac-secret`):** если сервер и агент
  настроены на РАЗНЫЕ значения `update_hmac_secret` (опечатка, неполный роллаут
  на часть флота), HMAC-проверка на агенте детерминированно и молча падает —
  `state="failed", reason="подпись пакета не сошлась"`. Это неотличимо от
  реальной атаки (fail-closed — корректно, но диагностика непрозрачна:
  оператор не может понять «я забыл обновить конфиг» vs «кто-то подделывает
  манифест»). До этого фикса рассинхронизация была НЕВОЗМОЖНА (обе стороны
  всегда делили один ingest_token) — новое поле добавляет способ достичь
  этого состояния. Малоценно для MVP (единственный сервер, ручной роллаут),
  но при росте флота стоит добавить нессекретный key-id/версию в манифест +
  конфиг, чтобы агент мог залогировать «key-id сервера=B, мой=A» (рассинхрон)
  отдельно от «key-id совпадает, HMAC не сошёлся» (подделка).
- **`server/db.py` (4857 строк) далеко за пределами лимита CLAUDE.md (P3-9,
  структурное наблюдение из stoperrors.md, не кодовая задача):** файл вырос
  органически по мере добавления фич (метрики, print-аналитика, принтеры,
  pruning, core CRUD) и сейчас далеко за порогом <800 строк/файл. Разбиение
  по зонам ответственности — самостоятельная, отдельно спланированная задача
  (по духу — R4: «shared/schema.py-смежный контракт», требует brainstorm/Plan
  в `docs/superpowers/` перед исполнением), намеренно НЕ включена в
  bugfix-проход stoperrors.md: смешивание крупного рефакторинга с точечными
  багфиксами по этому же файлу (P0-6, P0-7, P0-8-смежные части, P2-1, P2-15,
  P3-1 — все трогали `db.py` в рамках этого же прохода) резко повышает риск
  конфликтов и регресса — прямое нарушение «не навреди». Кандидат на будущий
  спринт после завершения stoperrors.md (весь план P0-P3 закрыт 2026-07-23).
