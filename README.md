# SRP — система раннего предупреждения отказов

Раннее предупреждение об отказах и деградации обычных офисных ПК под Windows.
Тонкий агент собирает телеметрию на каждой машине и отправляет её на сервер;
сервер хранит историю, считает оценки здоровья и байесовский риск отказа и
показывает парк машин на веб-дашборде.

Главный тезис: **абсолютные значения — слабый сигнал, информация живёт в
производных** (тренды, базовые линии, накопленная история). Агент только
сообщает, что наблюдал; всю аналитику делает сервер.

---

## Архитектура

**Тонкий агент / толстый сервер.** Агент не принимает решений — он собирает
четыре типа сообщений и шлёт их в общем конверте (`Envelope`):

| Тип          | Что это                                   | Как часто (по умолчанию) |
|--------------|-------------------------------------------|--------------------------|
| `inventory`  | Медленная «личность» машины (железо, ОС)  | раз в сутки              |
| `historical` | Скан «день-1»: прошлое машины как датасет | раз в сутки              |
| `heartbeat`  | Живые показатели (CPU, память, диск...)   | раз в 5 минут            |
| `events`     | Белый список событий журнала Windows      | раз в 15 минут           |

Сервер на каждое сообщение пересчитывает оценки:

- **4 оценки «день-1»** (0–100, выше = здоровее): `performance`, `reliability`,
  `wear`, плюс `risk_exposure` (0–100, выше = ХУЖЕ).
- **Байесовский риск по классам отказа** (накопитель, батарея,
  питание/перегрев, память, стабильность). Работаем в лог-оддсах:
  `posterior = sigmoid(prior + Σ весов признаков)`. Каждый член подписан —
  результат объясним по построению (дашборд показывает, какие признаки подняли
  риск). Веса заданы вручную и пока не калиброваны.

## Структура проекта

```
srp/
├── shared/schema.py        контракт сообщений (pydantic v2, единый источник истины)
├── server/
│   ├── main.py             сборка FastAPI-приложения  (python -m server.main)
│   ├── api.py              REST: /api/v1/ingest, /devices
│   ├── pipeline.py         приём конверта -> запись -> пересчёт оценок
│   ├── db.py               SQLite-хранилище
│   ├── scoring/            scores.py (4 оценки) + bayesian.py (риск)
│   ├── web/dashboard.py    серверный рендер (Jinja2, autoescape вкл.)
│   └── config.json         host / port / db_path / лимиты хранения
├── client/                 АГЕНТ — чистый стандартный Python, без зависимостей
│   ├── agent.py            главный цикл  (python -m client.agent)
│   ├── transport.py        отправка + офлайн-буфер (urllib, JSONL)
│   ├── collectors/         сбор телеметрии через PowerShell
│   └── config.json         server_url (IP сервера) + интервалы
├── requirements.txt        зависимости СЕРВЕРА
├── requirements-dev.txt    + pytest, httpx
└── smoke.py                быстрый E2E без сети (TestClient)
```

## Требования

- **Сервер:** Python 3.9+ и пакеты из `requirements.txt`
  (FastAPI, uvicorn, pydantic v2, Jinja2). SQLite встроен в Python.
- **Агент:** только Python 3.9+ — **сторонних пакетов не требуется** (urllib,
  subprocess, json, winreg из стандартной библиотеки). На офисных ПК Python
  часто не установлен — для массового развёртывания соберите `.exe`
  (например, PyInstaller), это вне рамок MVP.

## Запуск сервера

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
python -m server.main
```

Сервер слушает `0.0.0.0:8000` (настраивается). Откройте дашборд:
`http://<IP-сервера>:8000/`. REST живёт под `/api/v1/`.

Настройки — `server/config.json`:

```json
{ "host": "0.0.0.0", "port": 8000, "db_path": "srp.db",
  "retain_heartbeats": 500, "retain_events": 1000 }
```

## Запуск агента

```powershell
# один проход всех сборщиков и выход (для проверки/диагностики)
python -m client.agent --once

# постоянная работа по интервалам из config.json
python -m client.agent

# разовое переопределение адреса сервера
python -m client.agent --server http://212.42.56.189:8000 --once
```

**IP сервера задаётся в `client/config.json`** (поле `server_url`).
По умолчанию — `http://212.42.56.189:8000` (глобальный адрес).

```json
{ "server_url": "http://212.42.56.189:8000",
  "device_id": "",
  "heartbeat_interval_sec": 300, "events_interval_sec": 900,
  "inventory_interval_sec": 86400, "historical_interval_sec": 86400,
  "http_timeout_sec": 15, "buffer_path": "buffer.jsonl" }
```

`device_id` оставьте пустым — при первом запуске агент возьмёт стабильный
`MachineGuid` из реестра (или сгенерирует UUID) и запишет его обратно.
Если сервер недоступен, конверты копятся в `buffer.jsonl` и досылаются
(FIFO) при следующем успешном контакте.

## Безопасность

- Серийные номера дисков **хешируются (SHA-256)** в агенте; сырой серийник
  удаляется до того, как payload покинет процесс — сервер видит только
  `serial_hash`.
- Jinja2 autoescape включён: строки из телеметрии (hostname, текст события)
  экранируются — хранимого XSS из телеметрии нет.
- Все SQL-запросы параметризованы.
- Входные данные валидируются на границе через pydantic-модели.
- Секретов в коде нет.

> MVP-замечание: канал `/api/v1/ingest` пока без аутентификации — для боевого
> развёртывания добавьте общий токен/mTLS и HTTPS на reverse-proxy.

## Известные ограничения (осознанные для MVP)

- **Языконезависимость.** Используем CIM-классы `Win32_PerfFormattedData_*`, а
  не `Get-Counter`: пути счётчиков локализованы (на русской Windows английские
  пути не работают). Уровни событий маппим по числовому `$e.Level`, а не по
  `LevelDisplayName` (тоже локализован).
- **Задержка диска.** `AvgDisksecPerRead/Write` в форматированном классе —
  целое число, поэтому суб-секундная задержка усекается до 0 (здоровый диск =
  0 = нет ложной тревоги; всплывают только многосекундные зависания).
- **Троттлинг.** `cpu_perf_pct` (% Processor Performance) — прокси троттлинга,
  на холостом ходу естественно низкий (SpeedStep); значим только под нагрузкой.
- **SMART.** `Get-StorageReliabilityCounter` на части десктопов пуст — это
  нейтральный пробел (None), а не ноль; машина не выглядит здоровой ложно.
- **Веса риска** заданы вручную и не калиброваны — заглушка до появления
  размеченных отказов и survival-модели + изотонической калибровки.

## Разработка и качество кода

Проект следует принципу «зрелый старт»: инструменты качества настроены до того,
как код разрастётся. Вся конфигурация инструментов живёт в одном месте —
[`pyproject.toml`](pyproject.toml). Единая точка входа — `Makefile`.

| Команда           | Что делает                                                              |
|-------------------|-------------------------------------------------------------------------|
| `make install`    | Ставит зависимости для разработки (`requirements-dev.txt`)              |
| `make lint`       | Линт `ruff check .` (стиль, импорты, баги, упрощения)                   |
| `make format`     | Автоформат `ruff format .` + автоисправления `ruff check --fix`         |
| `make typecheck`  | Статическая типизация `mypy` (с плагином pydantic) по `shared`/`server` |
| `make security`   | Анализ безопасности `bandit` по `server shared client`                  |
| `make test`       | Юнит- и интеграционные тесты `pytest`                                   |
| `make coverage`   | Тесты с покрытием (term + `coverage.xml`); порог — 80 %                 |
| `make check`      | Полный гейт: lint → typecheck → security → coverage                     |
| `make clean`      | Удаляет кэши инструментов и артефакты сборки                            |

Без `make` (например, на чистой Windows) — те же команды напрямую:

```powershell
pip install -r requirements-dev.txt
ruff check .
mypy
bandit -c pyproject.toml -q -r server shared client
pytest --cov=server --cov=shared --cov-report=term-missing
```

### Инструменты

- **ruff** — линтер и форматтер. Набор правил: `E,W,F,I,B,C4,SIM`. Правила `UP`
  намеренно не включены (сохраняем явный `Optional[...]` и совместимость с
  Python 3.9); безопасность отдана `bandit`, поэтому группа `S` в ruff не дублируется.
- **mypy** — строгая типизация для аналитического ядра (`shared`, `server`),
  с плагином `pydantic.mypy`.
- **bandit** — поиск уязвимостей. Осознанные исключения помечены инлайн
  `# nosec <код>` с пояснением в комментарии рядом (PowerShell со статическим
  argv, привязка к `0.0.0.0`, SQL с литеральными именами таблиц).
- **pytest + pytest-cov** — тесты. Покрытие считается по `server` и `shared`
  (аналитическое ядро); клиент-агент проверяется живым E2E (`smoke.py`).
  Минимальный порог покрытия — **80 %**.
- **pre-commit** — те же ruff/mypy/bandit как git-хуки. Установка:
  `pre-commit install`.

### CI

[GitHub Actions](.github/workflows/ci.yml) запускается на каждый push в
`main`/`master` и на каждый pull request. Два независимых задания:

- **quality** (`ubuntu-latest`): `ruff` → `mypy` → `bandit` — платформо-независимые гейты.
- **test** (`windows-latest`, матрица Python 3.9 / 3.11 / 3.12): `pytest` с
  покрытием. Тесты идут на Windows осознанно — это реальная целевая платформа
  агента (winreg, PowerShell). Отчёт `coverage.xml` сохраняется как артефакт;
  опциональная загрузка в Codecov закомментирована и включается одной строкой.
