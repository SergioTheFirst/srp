# Сеть — фаза 2: карта сети + ось network_risk + аномалии подсетей

**Дата:** 2026-06-10
**Статус:** дизайн (производный от утверждённого обзора `2026-06-10-network-tools-design.md` §8; решения зафиксированы здесь)
**Объём:** только сервер. Агент и контракт (`shared/schema.py`) **не меняются** — фаза 1 уже отдаёт все нужные данные.

---

## 1. Цель

Фаза 1 научила агентов собирать сетевые данные. Фаза 2 заставляет сервер ими пользоваться:

1. **Карта сети** — кто есть в сети, сгруппировано по шлюзам; чужие устройства (без агента) видны по ARP-соседям.
2. **Ось `network_risk`** (Score100) — здоровье сети конкретной машины: потери/задержка до шлюза, APIPA, слабый Wi-Fi.
3. **Аномалии подсетей** — «вся подсеть теряет пакеты → проблема инфраструктуры (свитч/роутер), не ПК» — гасит массовые ложные тревоги.
4. **`network` становится доменом доверия** (`DOMAIN_SOURCES`) с семантическим валидатором — источник теперь решающий (питает ось), материальность требует проверки.

## 2. Ключевые решения (с обоснованием)

| # | Решение | Почему |
|---|---------|--------|
| D1 | Карта = **серверный рендер** (Jinja2), кластеры по **шлюзу**; никакой JS-библиотеки графов | Правило проекта «vanilla-JS, без SPA»; autoescape защищает; L3-топология офиса И ЕСТЬ звезда вокруг шлюза; в фазе 1 нет prefix/iface у соседей → честнее группировать по шлюзу наблюдателя |
| D2 | Сопоставление узлов по **MAC** (нормализованный `AA-BB-…`); сосед с MAC адаптера известного агента = эта машина | MAC стабильнее IP (DHCP); IP-фолбэка нет — лучше «неизвестное устройство», чем ложное слияние |
| D3 | OUI-вендор = **крошечный seed-словарь** (~10 виртуализация+RPi, уверенность 100%), остальное «неизвестно» | UNKNOWN важнее ложной уверенности — выдуманный вендор хуже отсутствия; паттерн `_KNOWN_BAD_FIRMWARE` («hook, не платформа»), курируется вне кода |
| D4 | `network_risk`: ведущий сигнал = **качество до шлюза** (loss/latency), APIPA = сильный самостоятельный, Wi-Fi-сигнал = слабый | Шлюз — единственная цель, обязанная отвечать; APIPA = фактический отказ сети; Wi-Fi-сигнал часто отсутствует |
| D5 | **ICMP-неоднозначность**: 100% потерь на ВСЕХ целях машины → не тревога, а «ICMP блокируется» (missing_evidence), качество = отсутствует | Фаервол ≠ отказ сети; полные потери только до DNS при живом шлюзе → тоже не тревога (DNS-серверы часто фильтруют ICMP) |
| D6 | Доверие: домен `{"network": {required:["network"]}}`; валидатор = stateless диапазоны (loss 0..100, latency 0..60000, signal 0..100); gate-fail домена → ось withheld | Materiality governor: источник стал решающим; диапазоны ловят мусор без state |
| D7 | Аномалия подсети = **read-side** (страница карты + аннотация на странице устройства), НЕ в `recompute_scores` | Кросс-устройство в per-device rescore = O(n²) на ingest (урок W4.0); человек смотрит страницы редко |
| D8 | Аномалия не меняет значение `network_risk` устройства — только **аннотация** оператору | Машина реально испытывает потери (ось честна); меняется атрибуция причины, это текст для человека |
| D9 | Порог аномалии: в кластере ≥2 отчитавшихся агентов И ≥60% с потерями ≥20% до шлюза | Паттерн fleet_anomaly (мин. когорта 2, malой выборке — осторожные пороги) |
| D10 | Тренд `gateway_latency` в trajectory: **direction-only** (как boot_time), без ETA | У задержки нет «порога отказа»; рост = деградация линка/свитча — повод смотреть |
| D11 | Уверенность оси ≤ **medium** | Один наблюдатель, ICMP груб; здоровый шлюз ≠ здоровая сеть за ним |
| D12 | Наблюдаемость (observability) теперь делит на 7 доменов: старый агент без `network` теряет ~14% | Честно: не видим сеть = слепое пятно; все агенты фазы 1 уже отдают network |

## 3. Что НЕ входит (анти-цели фазы 2)

- Активный скан / ping-перебор — **фаза 3, за гейтом**.
- Изменения агента/контракта; новые msg_type; новые таблицы БД (читаем JSON payload последнего `historical`).
- Граф-визуализация с физикой/зумом; websocket/SPA; авто-poll карты (данные меняются с каденсом `historical`).
- Полная OUI-база (мегабайты) — seed + курирование вне кода.
- Кросс-линки TCP-связей на карте (фаза-2.1 при реальной потребности; данные уже собираются).
- `network` в байесовские классы / required-домены health-осей дня-1 — ось самостоятельна.
