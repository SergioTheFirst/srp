# Трей-клиент SRP + установка одной командой + справочник организаций — Design Spec
_Дата: 2026-06-12, **v2** (уточнения пользователя: установка одной командой со всеми
параметрами; справочник орг/подразделений на сервере с расшифровкой в дашборде и
отчётах; после истечения сертификата — 1 уведомление в день, 7 дней).
Статус: на утверждении. Реализация — после одобрения, по этапам (TDD)._

## Цель
Превратить «невидимого» агента SRP в удобного клиента, который живёт в трее Windows:
напоминает об истечении личного сертификата (каждые 4 часа до истечения; после
истечения — раз в день в течение 7 дней), показывает IP и состояние машины, защищён
паролем, считает напечатанные страницы даже там, где журнал печати недоступен, и
ставится из сетевой папки **одной командой без вопросов** (~60 секунд), в которой уже
заданы пароль, коды организации/подразделения и адрес сервера. Сервер расшифровывает
коды в полные названия в дашборде и отчётах.

## 1. Архитектура: «две плоскости» (выбранная стратегия)

```
┌─ C:\SRP ──────────────────────────────────────────────┐
│ srp-agent.exe   ← SYSTEM, scheduled task (КАК СЕЙЧАС) │
│   собирает телеметрию, шлёт на сервер,                │
│   пишет status.json (атомарно, раз в цикл)            │
│                                                       │
│ srp-tray.exe    ← в сессии КАЖДОГО пользователя       │
│   (HKLM\...\Run), иконка в трее, читает status.json,  │
│   сам проверяет личный сертификат пользователя        │
│                                                       │
│ config.json · status.json · print_state.json · логи   │
└───────────────────────────────────────────────────────┘
```

Почему так, а не иначе (рассмотренные альтернативы):
- **Монолит в сессии пользователя** — отвергнут: теряем SYSTEM-права (SMART/WMI →
  trust UNKNOWN), сбор умирает при logoff, пользователь может убить процесс.
- **UI с сервера (toast/браузер)** — отвергнут: из session 0 уведомления в сессию
  пользователя не доставляются by design; пользователь явно просил трей.
- **Две плоскости** — стандарт всех корпоративных агентов (EDR/RMM): сбор живёт под
  SYSTEM всегда, UI живёт там, где пользователь. Падение трея не трогает сбор.

Ключевой факт, который делает разделение обязательным: **личный сертификат лежит в
`Cert:\CurrentUser\My` пользователя**. Агент под SYSTEM видит только хранилище
SYSTEM (W1.2 собирает LocalMachine + CurrentUser _системы_). Проверять сертификат
может ТОЛЬКО процесс в сессии пользователя — т.е. трей.

### IPC: один файл, одна сторона
`status.json` — агент пишет (tmp + `os.replace`), трей читает. **Канала команд
трей→агент НЕТ** (YAGNI: единственная мыслимая команда «собери сейчас» не нужна при
heartbeat раз в 5 минут; нет канала — нет поверхности атаки и валидации). Спул данных
этапа 8 (метаданные сертификатов) — отдельное гейтированное решение, не канал команд.
Свежесть: `age < 15 мин` → агент жив; иначе «служба не отвечает».

`status.json` (без секретов: ни токена, ни URL с кредами — редакция как в agent.py):
```json
{
  "ts": 1781234567, "agent_version": "0.3.0",
  "last_send_ok_ts": 1781234500, "last_send_error": "", "buffer_depth": 0,
  "hostname": "PC-042", "ips": ["192.168.1.42"],
  "org_code": "101", "dept_code": "7",
  "print_today_pages": 14, "print_month_pages": 312, "print_mode": "events",
  "disk_free_gb": 41.2, "uptime_days": 3.5
}
```

## 2. Трей: что видит пользователь

### Иконка (3 состояния, worst-of)
| Цвет | Условие |
|---|---|
| зелёная | всё ок |
| жёлтая | серт ≤ 30 дн · ИЛИ нет связи с сервером > 1 ч · ИЛИ агент молчит |
| красная | серт ≤ 7 дн / истёк без преемника · ИЛИ агент мёртв > 1 сут |

Tooltip: `PC-042 · 192.168.1.42 · всё в порядке`. Чистая функция
`icon_state(cert_state, agent_age, link_age) -> ok|warn|alert` — юнит-тесты.

### Панель (клик по иконке; одна страница, по-русски)
1. **Организация / отдел** — `org_code` / `dept_code` (+ имя ПК). Трей показывает
   **код, не название** — by design: клиент названий не получает (см. §7), поддержка
   слышит код по телефону и видит полное название у себя в дашборде.
2. **IP в локальной сети** — живое значение из `socket` (UDP-connect к серверу →
   `getsockname`; fallback `getaddrinfo`), только RFC1918; работает даже при мёртвом агенте.
3. **Сертификат** — «истекает через N дн. (15.07.2026)» / «истёк 03.06.2026» /
   «действителен до …» / «не найден»; список валидных и истёкших ≤ 90 дн.
4. **Связь** — «агент работает, отправка 14:02 ✓» / «нет связи с сервером с …».
5. **Печать** — «сегодня 14 стр. · за месяц 312 стр.» (+ пометка режима учёта).
6. **Диск C:** — «свободно 41 ГБ»; предупреждение < 10%.
7. **Без перезагрузки** — «3 дня»; мягкая подсказка при > 14 дн («перезагрузите ПК»).
8. **Версия агента** и телефон поддержки (`helpdesk_contact` из config).

Кнопки: **«Скопировать для поддержки»** (одна строка: ПК+IP+орг+серт+версия → буфер
обмена) · **«Обновить»**. Контекст-меню трея: Открыть панель · Обновить · О программе ·
**Выход (пароль)**.

### Технология (zero-deps инвариант сохранён)
- **Иконка/меню/балуны** — `ctypes` + win32 (`Shell_NotifyIcon`, `TrackPopupMenu`,
  скрытое окно с WNDPROC). Обязательно: обработка `TaskbarCreated` (перезапуск
  explorer → пере-добавить иконку). 3 статичных `.ico` (ok/warn/alert), без GDI-рисования.
- **Уведомление** = balloon (`NIF_INFO`) — Windows 10/11 сам рендерит его как toast.
  Никакого WinRT/COM (отвергнуто как оверинжиниринг).
- **Панель и парольный диалог** — `tkinter/ttk` **отдельным дочерним процессом**
  (`srp-tray.exe --panel`): нет конфликта message-loop трея и mainloop tkinter,
  падение панели не роняет иконку. PyInstaller бандлит tkinter сам.
- Один экземпляр — named mutex (`CreateMutexW`). Лог — `%LOCALAPPDATA%\SRP\tray.log`
  (RotatingFileHandler, как у агента).
- Код: `client/tray/` (≈5 файлов: `__main__.py`, `icon.py`, `panel.py`, `certs.py`,
  `state.py`), вся логика — чистые функции, ctypes-обёртка тонкая.

## 3. Сертификаты: движок напоминаний (v2)

Проверка: PowerShell 5.1 из трея (переиспользуем `client/collectors/ps.py`):
`Get-ChildItem Cert:\CurrentUser\My | ? HasPrivateKey` → JSON c **epoch-секундами**
(`[DateTimeOffset]::…ToUnixTimeSeconds()` — .NET 4.6+/PS 5.1; никаких локализованных
дат — инвариант языконезависимости). Пересчёт раз в 30 мин + при входе в систему.

### Группировка по Subject — защита от ложного шума
Сертификаты группируются по нормализованному Subject CN. Продление ЭЦП — это новый
сертификат **того же субъекта**: появление валидного преемника **гасит все
напоминания** о старом (и до, и после истечения) и показывает однократный
инфо-балун «Обнаружен новый сертификат, действует до 24.06.2027». Разные субъекты
(разные роли/системы) отслеживаются независимо — продление одного не маскирует
истечение другого.

### Расписание (на группу субъекта; чистая функция `should_nag(now, group, state)`)
| Состояние лучшего сертификата группы | Поведение |
|---|---|
| валиден, > 30 дн | тишина |
| валиден, 8–30 дн | балун каждые `tray_notify_hours` (4 ч), жёлтая иконка |
| валиден, ≤ 7 дн | балун каждые 4 ч, красная иконка, текст жёстче |
| **истёк, ≤ 7 дн назад** | **1 балун в календарные сутки**, красная иконка |
| истёк, > 7 дн назад | балуны прекращаются; панель и красная иконка продолжают показывать |

Анти-спам переживает перезапуски/relogin: `%LOCALAPPDATA%\SRP\tray_state.json` =
`{thumbprint: {last_nag_epoch, last_nag_date, new_cert_announced}}`. «Раз в сутки» =
по календарной дате (первая проверка нового дня), не скользящие 24 ч.

Тексты: «Сертификат истекает через 12 дн. (24.06.2026). Обратитесь в поддержку: …» /
«Сертификат ИСТЁК 03.06.2026. Работа с подписью невозможна. Поддержка: …».

Опция `tray_require_cert=false` (config): если `true` и валидного сертификата нет
вообще (включая пустое хранилище) — ежедневный балун «личный сертификат не
установлен» (для организаций, где ЭЦП обязательна каждому).

Новые поля config (additive): `tray_cert_warn_days=30`, `tray_notify_hours=4`,
`tray_require_cert=false`, `helpdesk_contact=""`.

## 4. Пароль

Уже есть инфраструктура (`config_password_hash`, PBKDF2-HMAC-SHA256 260k,
`hash_password`/`verify_password`) — не изобретаем вторую. Трей использует её же:
- Пароль требуется на: **Выход из трея**, изменение настроек, отключение
  напоминаний > 24 ч. Просмотр панели — без пароля (там нет секретов).
- 3 неверных подряд → блокировка ввода 5 мин (timestamp в tray_state.json).
- Задаётся параметром установки `--password` (в config попадает только hash;
  plaintext не пишется никуда).
- **Честные рамки** (фиксируем в доке и в «О программе»): это барьер от случайного
  закрытия обычным пользователем. Локальный администратор может убить процесс /
  задачу — защита от админа не заявляется. Реальная защита данных — ACL на C:\SRP.

## 5. Печать: основной путь + надёжный fallback

Сейчас: Event 307 `PrintService/Operational` (включаемый журнал). Требование: если
журнал включить не удалось — считать иначе.

**Решение — авторежим в каждом sweep, без поля в конфиге:**
```
IsEnabled? (Get-WinEvent -ListLog …PrintService/Operational → bool, локале-безопасно)
  ├─ true  → режим "events"  (существующий сборщик 307: страницы/принтер/пользователь)
  └─ false → режим "counter" (fallback)
```
- **Fallback**: CIM `Win32_PerfFormattedData_Spooler_PrintQueue` (соответствует
  инварианту «CIM Win32_PerfFormattedData_*, не Get-Counter»), по каждой очереди
  `TotalPagesPrinted` (накопительный с момента старта спулера). Агент хранит
  baseline в `print_state.json`, шлёт **дельты**; `current < baseline` → спулер
  перезапущен → baseline = current (потеря ограничена страницами между опросами,
  двойного счёта нет). Виртуальные принтеры — тот же фильтр `_VIRTUAL` по Name.
- Строки fallback: `{printer, pages=дельта, ts=время sweep, job_id=null,
  user_name=null, source="counter"}` — серверный dedup `UNIQUE(device_id, job_id)
  WHERE job_id IS NOT NULL` их не трогает; аналитика `/print` суммирует как обычно.
- Контракт: PrintJob + опциональное поле `source` ("events"|"counter") — additive,
  **без bump CONTRACT_VERSION**; колонка в БД additive. Честность: в counter-режиме
  нет пользователя/документов — БД хранит источник, дашборд может пометить.
- **Самовосстановление**: журнал включили позже (GPO/админ) → следующий sweep сам
  переключится на events (baseline сбрасывается, `last_sweep_ts=now` — без двойного
  счёта). Установщик включает журнал (`wevtutil sl … /e:true`) best-effort и пишет
  итог в отчёт установки; агент НЕ зависит от успеха установщика.
- Оба метода считают страницы, **отправленные на принтер**. Подтверждение «физически
  вышло» возможно только опросом счётчика самого принтера (SNMP) — сознательно НЕ в
  клиенте; возможная серверная сверка позже (netmap уже знает принтеры по OUI).
- Счётчики для панели: агент ведёт скользящие «сегодня/месяц» в `print_state.json`
  (date-keyed, prune) и кладёт в `status.json`.

## 6. Установка: ОДНА команда, ноль вопросов

Опора на видение из «деплой агента.txt» (PyInstaller, C:\SRP, тихий режим) + v2.

### Сборка (на машине разработчика, один раз на релиз)
`build.bat` → PyInstaller: `srp-agent.exe` (onedir, console), `srp-tray.exe`
(onedir, windowed, тот же payload-каталог), `srp-setup.exe` (onefile, console,
манифест `--uac-admin` — UAC поднимает сам). PyInstaller — **build-зависимость**
(`requirements-build.txt`), кодовая база `client/` остаётся чистым stdlib.
**onedir, не onefile** для агента/трея: мгновенный старт (нет распаковки во temp),
меньше ложных срабатываний AV, robocopy-дружелюбные дельта-обновления.

### Раскладка сетевой папки
```
\\server\srp$\            ← скрытая шара; ACL: ТОЛЬКО группа ИТ (в BAT — пароли!)
  setup.exe               ← установщик
  install-101-7.bat       ← готовые BAT по подразделениям (генерит дашборд, §7)
  payload\                ← onedir: srp-agent.exe, srp-tray.exe, DLL, ico
  config.template.json    ← политика организации: интервалы, tray_cert_warn_days,
                            tray_notify_hours, helpdesk_contact, site_code
  VERSION                 ← маркер версии для идемпотентного апгрейда
```
Граница «параметр vs шаблон»: **параметры команды = то, что отличается между
машинами/отделами; шаблон = политика, одинаковая для всех**.

### Одна команда (BAT с параметрами)
```bat
@echo off
\\server\srp$\setup.exe --server http://192.168.1.10:8000 ^
  --org 101 --dept 7 --password "Zx9!..." --token "mysecret"
```
Техник запускает BAT своего подразделения (двойной клик → UAC) — **ноль вопросов**.

| Параметр | Обяз. | Назначение |
|---|---|---|
| `--server URL` | да* | адрес сервера (*может прийти из config.template.json) |
| `--org CODE` | да | код организации, `^[A-Za-z0-9_-]{1,16}$` (рекомендация — числа) |
| `--dept CODE` | нет | код подразделения, тот же формат |
| `--password X` | нет | пароль трея/конфига → PBKDF2-hash |
| `--token X` | нет | ingest_token (X-SRP-Token, как у сервера) |
| `--helpdesk "…"` | нет | контакт поддержки (перекрывает шаблон) |
| `--comment "…"` | нет | свободная метка машины |
| `--no-tray` | нет | без UI — серверы/киоски (Run key не ставится) |
| `--allow-offline` | нет | ставить, даже если сервер сейчас недоступен (агент буферизует) |
| `--quiet` | авто | включается сам, когда заданы --server и --org |
| `--uninstall [--purge]` | — | удаление (+ данные/логи по --purge) |

Заданы все обязательные → тихий режим автоматически; запуск `setup.exe` совсем без
параметров остаётся интерактивным fallback (спросит сервер и орг-код).

### Шаги установки (цель < 60 c)
1. UAC (манифест) → 2. `robocopy payload\ C:\SRP\` (/MT; VERSION совпала →
пропуск, повторный запуск ≈ мгновенный) → 3. **ACL**: `icacls C:\SRP /inheritance:r
/grant SYSTEM:F Administrators:F "Users:RX"` (дефолт корня C:\ даёт обычным
пользователям ЗАПИСЬ — закрываем; это и есть настоящая защита config.json, пароль —
UI-слой) → 4. config.json = merge(template, параметры), device_id и прежние значения
сохраняются, UTF-8 **без BOM** → 5. журнал печати `wevtutil … /e:true` (best-effort)
→ 6. валидация `srp-agent.exe --once` (живой сбор+отправка; провал → откат, внятная
ошибка; с `--allow-offline` шаг мягкий) → 7. scheduled task из **встроенного XML**
(`schtasks /create /xml`: AtStartup, SYSTEM, Highest, restart 3×1мин, без лимита —
паритет с текущим install-service.ps1, но без PowerShell → нет execution policy) +
`HKLM\…\Run\SRP-Tray` (кроме --no-tray) → 8. старт задачи + запуск трея в текущей
сессии → 9. отчёт по-русски: сервер ✓, журнал печати events/counter, орг=101/отдел=7,
трей запущен.

### Диагностика для RMM/GPO
- **Коды возврата**: 0 = ок · 2 = неверные/неполные параметры · 3 = сервер недоступен
  (без --allow-offline) · 4 = копирование/ACL · 5 = регистрация задачи/автозапуска ·
  1 = прочее. Массовый раскат: `psexec \\PC -s \\server\srp$\install-101-7.bat`,
  GPO Startup Script, любой RMM — по exit code видно итог.
- **`C:\SRP\install.log`** — полный протокол каждой установки/апгрейда (append).

`setup.exe --uninstall`: стоп задачи, taskkill srp-tray, удалить task + Run key;
файлы/данные — по `--purge`. Апгрейд = повторный запуск того же BAT с новой версией
в шаре (стоп → robocopy дельта → старт; config не трогается).

### Безопасность команды
Пароль/токен лежат в BAT на шаре и мелькают в process list. Митигация: шара скрытая
(`$`), ACL «только группа ИТ» (это деплой-инструмент, не публичная папка); на особо
чувствительных хостах — ставить без `--password/--token` и вписать в config.json
вручную (путь из заметки пользователя). `--password-stdin` отвергнут (оверинжиниринг
при шаре под ACL).

## 7. Сервер: справочник организаций и подразделений

Коды задаются в **настройках сервера** и расшифровываются в полные названия в
дашборде и отчётах. Клиент названия **никогда не получает и не хранит** (решение
identity-labels): в телеметрии — только коды.

### Носитель: файл настроек, не админ-UI
`org_directory.json` рядом с конфигом сервера (путь — `ServerConfig` /
env `SRP_ORG_DIRECTORY`):
```json
{ "organizations": [
    { "code": "101", "name": "ООО «Ромашка»",
      "departments": [ { "code": "7",  "name": "Бухгалтерия" },
                       { "code": "12", "name": "Склад" } ] } ] }
```
- Загрузка при старте + автоперечитывание по mtime (правка без рестарта).
  Битый JSON → ошибка в лог, продолжаем со старой копией (никогда не падаем).
- Почему файл, а не таблицы+UI: дашборд пока без аутентификации — UI-редактирование
  позволило бы любому в сети переименовывать организации; файл = та же модель
  доверия, что `ingest_token`; нет миграций; бэкап = копия файла.

### Расшифровка — render-time, единым helper'ом
Названия подставляются при отображении, в БД не пишутся → **переименование
организации мгновенно отражается во всех страницах и во всей истории**, ничего не
ломая. Где: fleet (группировка организация → отдел, фильтры с названиями),
страница устройства (заголовок), `/print` (графики по отделам с названиями),
CSV-экспорт печати (+ колонки `org_name`/`dept_name`), все будущие отчёты.

### Гигиена данных
- Код вне справочника → показываем код + чип **«нет в справочнике»** — ловит
  опечатки в BAT за минуты. Телеметрия НЕ отклоняется (UNKNOWN over rejection).
- **Унификация (один источник истины)**: свободнотекстовая `devices.department`
  (PATCH-редактируемая, по ней сейчас группирует /print) — **DEPRECATED**: /print и
  CSV переходят на `dept_code` + справочник (на переходный период
  `COALESCE(имя по dept_code, department, 'Без отдела')`); поле редактирования на
  device.html переключается на `comment`; колонка в БД остаётся (additive), из UI
  больше не пишется.

### Бонус-наблюдаемость флота (бесплатно — данные уже есть)
- Колонка «версия агента» + фильтр «устаревшие» (сравнение с VERSION актуального
  релиза) — видно, где забыли перезапустить установщик.
- Чип «новых машин за 7 дн» (по `first_seen`) — контроль раската.

### Генератор команды деплоя (опц., этап 7)
Страница «Развёртывание» в дашборде: выбираешь организацию/отдел из справочника →
получаешь готовый BAT-текст с правильными кодами и server_url, но с плейсхолдерами
`<ПАРОЛЬ>`/`<ТОКЕН>` (дашборд без auth — секреты в него не кладём). Убивает опечатки
кодов на корню; админ вставляет секреты и кладёт BAT в шару.

## 8. Безопасность / приватность (R4-поверхности)
- Закрытые ключи сертификатов **никогда** не читаются (инвариант W1.2) — только
  metadata: subject CN, issuer CN, thumbprint, not_after.
- status.json без секретов; токен живёт только в config.json под ACL.
- Все PS-вызовы — существующий `run_ps` (таймауты, JSON, числа/epoch/bool — никакого
  парсинга локализованного текста; флор PS 5.1: `-ListLog`, `Cert:`, `DateTimeOffset`).
- Трей НЕ ходит на сервер (токен ему не нужен) и НЕ может писать в C:\SRP.
- Шара деплоя: скрытая, ACL «только ИТ» (в BAT — секреты).
- PyInstaller-экзешники: риск SmartScreen/AV false-positive → onedir, console-сборки,
  при наличии серта — подпись кода; иначе GPO-исключение Defender.
- Пароль = сдерживание, не security boundary (см. §4) — заявляем явно.

## 9. Тесты (gate: cov ≥ 80%, ruff/mypy/bandit, smoke)
- Движок сертификатов: subject-группировка (преемник гасит старый; разные субъекты
  независимы), окна 30/7/0 дн, интервал 4 ч, **пост-истечение: ровно 1/календарный
  день × 7, затем тишина**, однократность инфо-балуна, tray_require_cert,
  персистентность анти-спама. Фикстуры: RU-subject, несколько сертов, без ключа.
- `icon_state` worst-of; freshness status.json.
- Fallback печати: дельта, перезапуск спулера (reset), фильтр виртуальных,
  автопереключение events⇄counter без двойного счёта.
- `status.json`: атомарность, отсутствие токена/кредов (пин-тест).
- Справочник: decode, неизвестный код (чип), mtime-reload, битый JSON → старая копия;
  /print COALESCE-переход; CSV-колонки названий.
- Setup: разбор параметров, авто-quiet, exit codes, merge config (чистые части).
- Контракт: `source` additive (старый сервер ↔ новый агент и наоборот).
- ctypes/tkinter-обёртки — тонкие адаптеры, ручной чек-лист (иконка, балун,
  TaskbarCreated, два пользователя RDP); вся ветвящаяся логика — чистые функции.

## 11. Отклонено осознанно (анти-оверинжиниринг)
WinRT toast через COM (балунов достаточно) · pystray/любые deps (запрет) · канал
команд трей→агент (нет команд) · HTTP-сервер в агенте для IPC (файла достаточно) ·
self-update (placeholder не активируем) · SNMP-опрос принтеров из клиента (сервер,
позже) · health-score в трее v1 (нужен бестокенный endpoint; и не пугать
пользователей — отложено) · watchdog трея (Run key восстановит при logon) ·
**админ-UI справочника** (дашборд без auth — файл безопаснее) · `--password-stdin`
(шара под ACL решает) · названия организаций в клиенте (коды — by design).

| Риск | Митигация |
