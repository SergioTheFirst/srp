# Мониторинг сетевых принтеров — дизайн (spec)

- **Дата:** 2026-06-17
- **Статус:** утверждён пользователем (подход A, без алертов в v1)
- **Маршрут:** R4 (новый модуль, ingest/SQL/dashboard surface) → spec → план → TDD по этапам → ревью сабагентом
- **План:** `docs/superpowers/plans/2026-06-17-printer-monitoring.md`

## 1. Цель и контекст
SRP уже собирает печать **по компьютерам** (`client/collectors/print_jobs.py` → страница `/print`, `/fleet/print/*`):
кто печатал, на каком принтере, сколько страниц, по отделам. Приоритет «какой ПК сколько печатает в месяц» **уже закрыт** — модуль его НЕ переписывает.

Новый модуль собирает телеметрию **с самих принтеров как устройств**: общий счётчик страниц, расходники, бумага/лотки,
ошибки, инвентарь (модель/серийник/прошивка) + обнаружение принтеров в сети. Это честно закрывает приоритет №2
(«какой принтер сколько бумаги тратит») — по **аппаратному счётчику принтера**, а не только по сумме заданий с ПК
(аппаратный счётчик ловит печать с машин без агента, копии, факсы, сканы).

## 2. В объёме / вне объёма
**В объёме (v1):**
- Обнаружение принтеров: тихое (по умолчанию) + активный скан (за стоп-гейтом, опц.).
- Опрос по приоритету: SNMP (Printer-MIB → HOST-RESOURCES-MIB → vendor OID) → IPP → HTTP/веб-интерфейс.
- Реестр вендор-драйверов (HP, Xerox, Kyocera, Canon, Brother, Lexmark, Ricoh, Epson + generic).
- Единая база OID, собранная из 4 репозиториев + RFC, дубли убраны.
- Сбор: общая информация, счётчики, расходники, бумага/лотки, ошибки/статус.
- Хранение во времени (append-only), страница `/printers`, сверка с `print_jobs` («бумага по принтерам»).

**Вне объёма (v1):**
- **Алерты/уведомления** (мало тонера, замятие) — пользователь отложил; отдельный будущий этап.
- Вплетение в trust/scoring — телеметрия **информационная** (как `user_certs`).
- Учёт по пользователям — уже есть в `print_jobs`, не дублируем.
- WMI через Windows Print Server, CUPS API, сырой JetDirect-парсинг — отмечены как будущие источники, в v1 не строим
  (SNMP/IPP/HTTP покрывают целевой парк; добавляются драйвером/источником без смены ядра).
- Любая запись в принтеры — **только чтение**: SNMP GET (никогда SET), только read-эндпоинты IPP/HTTP.

## 3. Ограничения (инварианты SRP)
- **Ядро опроса = чистый stdlib** (sockets/urllib/struct), ноль сторонних пакетов → переносимо в агент (агент = zero-dep).
- Любой агентский PowerShell (`Get-Printer`/`Get-PrinterPort`) — **WinPS 5.1 floor**, языконезависимо.
- **Активный скан — за письменным разрешением безопасности**, выключен по умолчанию (как сетевая Фаза 3).
- Приватность: наружу/в БД уходят только **RFC1918**-адреса принтеров. Серийник **принтера** = инвентарный атрибут
  устройства (НЕ персональные данные и НЕ серийник диска агента) → храним как есть; решение зафиксировано здесь.
- Сервер: SQL параметризован; Jinja2 autoescape ON; вывод любых строк принтера в JS-приёмник — через `srpEsc`.
- Контракт агента (`shared/schema.py`): новое поле обнаружения = **additive-optional → без bump CONTRACT_VERSION**.
- Серверный сбор пишет в БД напрямую, **минуя контракт агента** (контракт нужен только для агентских подсказок обнаружения).
- **UNKNOWN важнее ложной уверенности**: недоступный принтер → статус UNKNOWN, значения не выдумываем.

## 4. Архитектура (подход A)
Чистый stdlib-движок опроса, запуск **на сервере** (плоская сеть — сервер видит принтеры); тот же код позже
переносится в агент для удалённых подсетей **без переписывания**.

```
Источники обнаружения            Опрос (server/printers/)              Хранение            UI
─────────────────────            ────────────────────────             ─────────           ──────────────
порты спулера (агент) ┐
ARP-карта (есть)      ├─► registry ─► scheduler+ThreadPoolExecutor ─► probe ─► PrinterReading ─► DB tables ─► /printers
список инженера       ┘                 (таймауты/ретраи/кэш)        SNMP→IPP→HTTP            (append-only)   + сверка с print_jobs
активный скан (за гейтом, опц.) ┘                                    driver registry
```

**Размещение:** новый пакет `server/printers/`, **строго stdlib** (это и есть условие переносимости в агент).
Если позже понадобится агентский сбор — пакет повышается до верхнеуровневого/вендорится в `client/` без смены кода.

## 5. Компоненты (одна ответственность на модуль)
- `server/printers/snmp.py` — движок SNMP v1/v2c: GET/GETNEXT/GETBULK, BER/ASN.1 кодек, таймауты, ретраи с backoff,
  кэш статических OID, пакетная упаковка нескольких OID в один запрос (минимум round-trips).
- `server/printers/oids.py` — единая база OID (Printer-MIB RFC 3805, HOST-RESOURCES-MIB RFC 2790 + vendor), без дублей.
- `server/printers/drivers/` — реестр вендор-драйверов. Драйвер = декларативное сопоставление
  «опознать вендора (по `sysObjectID`/`sysDescr`) → какие OID читать на каждый показатель + парсер значения».
  Новый вендор = новый файл, **ядро не меняется** (паттерн alfonsrv/printer-monitoring).
- `server/printers/ipp.py`, `server/printers/http_probe.py` — fallback по приоритету после SNMP.
- `server/printers/classify.py` — «принтер ≠ ПК»: только при положительном признаке принтера
  (ответ Printer-MIB / IPP / printer-enterprise `sysObjectID`), не по одному открытому порту.
- `server/printers/discovery.py` — сведение источников в дедуплицированный список принтеров (ключ = серийник, иначе MAC, иначе IP).
- `server/printers/collector.py` — нормализация результатов в единый `PrinterReading`.
- `server/printers/scheduler.py` — интеграция с lifespan-loop сервера (как retention/purge sweep) + ThreadPoolExecutor.

## 6. Обнаружение
Приоритет (тихие методы — по умолчанию, разрешение безопасности НЕ нужно):
1. **Порты спулера агента** — агент читает локальную настройку печати (`Get-Printer`/`Get-PrinterPort` → IP/hostname порта).
   Это чтение конфига, не скан. Передаётся новым additive-полем контракта (без bump).
2. **ARP-карта** — уже строится сетевым коллектором/`server/analytics/netmap.py`.
3. **Список инженера** — IP/подсети в конфиге сервера.
4. *(За стоп-гейтом, опц.)* **активный скан**: SNMP-broadcast + перебор подсети + проба портов 161/9100/631/80/443/515.
   Выключен по умолчанию; включается только явным флагом-разрешением + письменным согласием безопасности.

## 7. Драйверы и база OID (расширяемость)
- **Стандарт-первым:** базовый профиль Printer-MIB + HOST-RESOURCES-MIB покрывает большинство принтеров.
- **Вендор-драйвер** добавляет/переопределяет OID там, где стандарт пуст или врёт, и несёт парсеры
  (напр. Brother — hex-строки расходников; Canon/Kyocera — таблицы счётчиков обходом).
- **Выбор драйвера:** прочитать `sysObjectID` → enterprise-префикс `1.3.6.1.4.1.<N>` → вендор → драйвер;
  иначе generic Printer-MIB. Затем при нехватке — IPP, затем HTTP.
- **База OID** собирается из репозиториев (alfonsrv/printer-monitoring — карта vendor-семейств; ynlamy/check_snmp_printer —
  стандартный Printer-MIB набор; torgeirl/printer_tools — парсинг расходников; vvalchev/printer-scanner — обнаружение)
  + RFC, дубли убираются. Это **Этап 0** плана (до кода движка — обязательное условие пользователя).

## 8. Хранение
Новые таблицы (append-only, как `historical`; миграции additive + backfill, как принято в `server/db.py`):
- `printers` — инвентарь (latest): ключ-идентичность, IP, hostname, MAC, vendor, model, serial, firmware, last_seen, last_status.
- `printer_readings` — снимки во времени: ключевые скаляры для быстрых выборок/трендов
  (`total_pages`, `color_pages`, `mono_pages`, `duplex_pages`, `status`, `received_at`) + полная нормализованная
  деталь (расходники, лотки, ошибки) валидированным **JSON-блобом** (как `user_certs` хранится JSON — без взрыва таблиц).
- Retention — счётчик-кап на принтер (как `_retain_hist`).
Идентичность принтера: серийник → иначе MAC → иначе IP (дедуп в `discovery.py`); смена IP при том же серийнике не плодит дубль.

## 9. Дашборд `/printers`
- SSR-страница (Jinja2 autoescape) + JSON-остров + Plotly (как `/print`); любой агент/принтер-текст в JS — через `srpEsc`.
- Карточки принтеров: полоски расходников (%, остаток), счётчики, ошибки/замятия/лотки, инвентарь.
- История счётчиков (тренд), и **сверка с `print_jobs`**: аппаратный счётчик принтера ↔ сумма заданий по этому принтеру.
- RU-текст оператору; машинные значения (статусы, ключи) — латиницей.
- nav-ссылка «принтеры» в `base.html`. API: `GET /api/v1/printers`, `GET /api/v1/printers/{id}`.

## 10. Производительность
- `ThreadPoolExecutor` — параллельный опрос многих принтеров; на каждый хост — таймаут и ретраи с backoff.
- Кэш статических OID (serial/model/firmware читаем редко; счётчики/расходники — каждый цикл).
- GETBULK для табличных OID; пакетная упаковка нескольких OID в один GET → минимум SNMP-запросов.
- Интервал опроса — в конфиге сервера; цикл в lifespan, аккуратная отмена (`contextlib.suppress(CancelledError)`).

## 11. Классификация «принтер ≠ ПК»
Хост считается принтером **только** при положительном признаке: ответ ветки Printer-MIB `1.3.6.1.2.1.43`,
или `hrDeviceType` = printer, или printer-enterprise `sysObjectID`, или IPP-идентификация принтера.
Один открытый порт (80/443) принтером НЕ делает.

## 12. Доверие/скоринг
Телеметрия принтеров — **информационная**, в trust/scoring v1 НЕ вплетается. Недоступный принтер → UNKNOWN
(не «0 страниц», не «всё ок»). Каждый снимок несёт, какой протокол сработал (SNMP/IPP/HTTP) и свежесть.

## 13. Безопасность / governance
- **Активный скан — за письменным разрешением** (стоп-гейт проекта); по умолчанию выключен, отдельный конфиг-флаг.
- SNMP — **только чтение** (GET/GETNEXT/GETBULK, никогда SET). IPP/HTTP — только read-эндпоинты.
- SNMP community по умолчанию `public`, версия v2c (fallback v1); оба переопределяются в конфиге; v3 отложен.
- Только RFC1918-адреса принтеров покидают сборщик/попадают в БД.
- `bandit`-чистота; сокеты с таймаутами; ввод от принтера = недоверенный → строгий парсинг + капы длин/количеств.

## 14. Тестирование (gate ≥80% cov)
- `snmp.py`: кодек на синтетических BER/PDU (round-trip), таймаут/ретрай, reset/edge-кейсы.
- `drivers/`: парсинг значений на зафиксированных ответах (включая Brother hex, Canon/Kyocera таблицы).
- `classify.py`: принтер vs ПК (положительные/отрицательные признаки).
- `discovery.py`: слияние/дедуп источников (серийник > MAC > IP).
- `db`: миграции additive + backfill идемпотентны; retention-кап.
- Дашборд: рендер 200, XSS-пин (враждебные model/hostname через `srpEsc`/autoescape), сверка с `print_jobs`.

## 15. Этапы (сводка; детали — в плане)
0. База OID из 4 репозиториев + RFC (дубли убраны). 1. stdlib SNMP-движок. 2. Драйвер-реестр + Printer-MIB →
`PrinterReading` + классификатор. 3. Тихое обнаружение. 4. Планировщик + таблицы БД. 5. Вендор-драйверы + IPP/HTTP.
6. Дашборд `/printers` + сверка. 7. *(за гейтом, опц.)* активный скан.
Каждый этап: ветка → TDD (RED→GREEN) → гейт зелёный → ревью сабагентом → `merge --no-ff`.
