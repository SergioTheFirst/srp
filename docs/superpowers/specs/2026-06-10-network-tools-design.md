# Сетевые инструменты SRP — обзор сети и сетевая аналитика

**Дата:** 2026-06-10
**Статус:** дизайн утверждён пользователем → готовится план реализации
**Эта спека покрывает в деталях ФАЗУ 1** (сбор сетевых данных агентом). Фазы 2 и 3 описаны как контекст-ориентир, у каждой будет своя спека.

---

## 1. Цель и зачем

Расширить SRP (ранняя диагностика отказов парка Windows-ПК) новой способностью — **видеть сеть** и **анализировать её**: какие машины есть в сети, как они связаны, где проблема инфраструктуры (свитч/линк/Wi-Fi), а не отдельного ПК.

Главная цель пользователя — **карта сети** и **«взять максимум из того, что агенты могут отдать»**, плюс заставить агентов собирать нужные сетевые данные.

### Декомпозиция на 3 фазы (утверждённый порядок)

```
Фаза 1  Агент собирает сетевые данные  (пункты 1 + 2)   ← ЭТА СПЕКА
   │      здоровье сети на машине · соседи (ARP) · локальные связи
   ▼
Фаза 2  Сервер: карта + аналитика флота (пункты 4 + 5)   (своя спека)
   │      union соседей со всех агентов · карта-визуализация ·
   │      ось риска "network_health" (Score100) · аномалии подсетей
   ▼
Фаза 3  Активный поиск устройств / скан (пункт 3)          (своя спека, за гейтом)
          выключен по умолчанию · только с разрешения · rate-limit
```

**Правило:** фаза N+1 не начинается, пока фаза N не закрыта по гейтам проекта (`make check` зелёный, smoke OK, ревью).

## 2. Что входит и НЕ входит в эту спеку

**Входит (фаза 1):**
- Новый коллектор агента `network` — собирает здоровье сети, соседей (ARP), локальные связи.
- Дополнение контракта (`shared/schema.py`) — новые **необязательные** поля в `HistoricalPayload`. Без смены `CONTRACT_VERSION`.
- Сервер хранит данные (уже умеет — JSON-payload `historical`), регистрирует источник `network` в слое доверия **без гейта**.
- Минимальный блок на странице устройства — чтобы видеть, что данные доходят (видимое изменение для CHANGELOG + ручной проверки).
- Тесты: golden-фикстуры коллектора, локаль-фикстуры, фильтр приватности, частичные сбои, контракт-совместимость, ungated-доверие.

**НЕ входит (явные анти-цели этой фазы):**
- Карта-визуализация и серверная аналитика — это **фаза 2**.
- Активный скан/ping-перебор подсети, проверка чужих портов — это **фаза 3**, за гейтом.
- **Имена программ** в связях — собираем только адрес+порт+состояние (порт уже называет услугу).
- **Внешние интернет-адреса** — агент выкидывает их **до отправки** (приватность).
- Маршрутные таблицы, история DNS-запросов пользователя, любой контент трафика.

## 3. Инварианты проекта, которые соблюдаем

- **Тонкий агент:** `client/` — чистая stdlib, ноль зависимостей. Сбор только через встроенный PowerShell/CIM (`urllib/subprocess/json` уже разрешены). Никаких nmap/scapy.
- **Язык-независимость:** только числовые поля и английские enum'ы CIM-классов (`Win32_PerfFormattedData_Tcpip_NetworkInterface`) и `Get-Net*`-команд; **никакого парсинга локализованного текста**. Прецедент уже в `client/collectors/heartbeat.py` (использует `Win32_PerfFormattedData_Tcpip_NetworkInterface`).
- **Приватность:** внешние адреса не покидают агента (как и сырой серийник диска — фильтруем в агенте). MAC/IP внутренней сети не хешируем — это инфраструктура, без них карта невозможна.
- **UNKNOWN важнее ложной уверенности:** нет данных / нет прав → источник `network` показывает `empty`/`blocked`, а НЕ «здорова». Сетевые данные в фазе 1 **не влияют** на скоры здоровья (ungated).
- **Контракт:** аддитивно, `extra="allow"` уже включён → старый сервер молча игнорирует новые поля, новый сервер терпит их отсутствие. Версия не растёт.
- **Код:** Python 3.9, `Optional` явный, строки ≤100, двойные кавычки, файлы <800 / функции <50 строк, ранние возвраты, иммутабельность. `mypy` теперь покрывает и `client/` → коллектор обязан типизироваться.

## 4. Данные, которые собираем (контракт)

Новые pydantic-модели в `shared/schema.py`, добавляются как **необязательные** поля в `HistoricalPayload` (рядом с `certificates: list[CertInfo]`). Имена — предварительные.

### 4.1 Здоровье сети машины

`NetAdapter` (по одному на сетевую карту):
- `name` (friendly, только для показа), `kind` (`ethernet`/`wifi`/`other` — из числового типа, не текста), `mac`, `up` (bool из `OperationalStatus`), `link_mbps`.
- IP-настройки: `ipv4` (список), `ipv6` (список), `prefix`/`mask`, `gateway`, `dns` (список), `dhcp` (bool).
- Wi-Fi (если есть): `ssid`, `signal_pct`, `channel`. **Только из язык-независимого числового источника** (CIM); если такого нет — `signal_pct`/`channel` опускаем, оставляем сам факт Wi-Fi-адаптера и `ssid` (это значение-данные, не локализованная подпись). `netsh wlan` (локализованный текст) **не использовать** — нарушит инвариант. См. открытый вопрос 6.
- Счётчики ошибок (из `Win32_PerfFormattedData_Tcpip_NetworkInterface`, числовые): `rx_errors`, `tx_errors`, `discards`.
- `ip_conflict` (bool) — если Windows сообщает о конфликте адреса.

`NetQuality` (качество связи, числа — основа будущих трендов в фазе 2):
- `target` (`gateway`/`server`/`dns`), `latency_ms` (медиана нескольких проб), `loss_pct`, `samples`.
- `dns_resolve_ms` — время ответа DNS на контрольное имя.

### 4.2 Данные для карты

`NetNeighbor` (из `Get-NetNeighbor` — то, что машина уже знает, без сканирования):
- `ip`, `mac`, `state` (числовой/enum `Reachable`/`Stale`/…), `iface`.

`NetConnection` (из `Get-NetTCPConnection`, **только внутренняя сеть**):
- `local_ip`, `local_port`, `remote_ip`, `remote_port`, `state` (enum), `direction` (in/out — выведено из портов).
- **Без** PID/имени процесса.

### 4.3 Ограничители объёма

- Списки `neighbors` и `connections` кап на снимок (предв. 256 каждый, top-N по свежести) — чтобы не раздувать строку `historical.payload` на флоте.
- Сервер для карты использует **последний снимок** на устройство; числовые метрики (`NetQuality`) копятся во времени (серия `historical`) для фазы-2 трендов.

## 5. Коллектор агента — `client/collectors/network.py`

Шаблон — `client/collectors/print_jobs.py` + CIM-паттерн из `heartbeat.py`.

### 5.1 Точка входа

```python
def collect_network() -> CollectorResult:
    # CollectorResult = (payload | None, source_health: dict[str, SourceHealth])
```

Регистрируется в `client/collectors/sources.py` как константа `NETWORK = "network"` рядом с `PRINT_JOBS`.

### 5.2 Источники данных (всё через `run_ps()` → JSON → `as_list()`)

| Блок | Команда / CIM-класс | Поля (числовые/enum) |
|------|--------------------|----------------------|
| Карты + ошибки | `Win32_PerfFormattedData_Tcpip_NetworkInterface` (CIM) | счётчики ошибок/отбросов |
| Карты (конфиг) | `Get-NetAdapter`, `Get-NetIPConfiguration` | статус, скорость, IP, шлюз, DNS |
| Wi-Fi | `Get-NetConnectionProfile`/CIM (SSID-значение) + числовой сигнал ТОЛЬКО если есть язык-независимый CIM-источник | SSID; signal/channel — опционально |
| Соседи | `Get-NetNeighbor` | IP, MAC, state |
| Связи | `Get-NetTCPConnection` | local/remote addr+port, state |
| Качество | `Test-Connection -Count N` (числовой `ResponseTime`/статус) | latency, loss |

`run_ps()` уже форсирует UTF-8 и парсит JSON; `as_list()` снимает квирк ConvertTo-Json (один объект vs массив); `_i()/_f()` — безопасное приведение чисел (как в `heartbeat.py`).

### 5.3 Фильтр приватности (в агенте, ДО отправки)

- «Внутренняя сеть» = RFC1918 (`10/8`, `172.16/12`, `192.168/16`) + адрес настроенного сервера SRP.
- `NetConnection`: отбрасываем, если `remote_ip` не внутренний (кроме адреса сервера). Loopback (`127/8`, `::1`) — отбрасываем (шум).
- `169.254/16` (APIPA): как ребро карты не используем, но `dhcp=false`-симптом «нет DHCP» допустимо отразить в здоровье карты.
- Внешние адреса **никогда не сериализуются** в payload — фильтр стоит до сборки модели.

### 5.4 Обработка ошибок и статус источника

- Каждый под-вызов PowerShell обёрнут отдельно: сбой одного блока → этот блок пустой и помечен, остальное всё равно уходит.
- Возврат `source_health[NETWORK]` со статусом по `field_status()`:
  - всё собрано → `ok`; часть блоков пуста → `partial`; `run_ps` вернул `blocked`/`timeout`/`absent` целиком → пробрасываем (через `failed([NETWORK], status)`).
  - **Никогда не возвращаем молчаливый `None` без статуса** — оператор должен отличать «нет данных» от «здорово».
- Коллектор не бросает исключений наверх; любые неожиданности → `partial` + причина.

### 5.5 Каденс

Сбор идёт в составе пакета `historical` (как сертификаты/печать) — **на той же периодичности, что и текущий `historical`**. Новый таймер не вводим.

## 7. Сервер — приём, хранение, доверие

### 7.1 Хранение (изменений по сути нет)

- `server/pipeline.py::ingest_envelope()` для `msg_type == "historical"` уже зовёт `db.store_historical(...)`, которая пишет **весь payload как JSON-блоб** (`server/db.py::store_historical`). Новые под-массивы сохраняются автоматически — **новая таблица не нужна** (паттерн сертификатов, не печати).
- Ретеншн — существующий кап `_retain_hist` на устройство.
- Последний снимок для карты — существующий «latest-by-id»; серия качества — существующая `get_historical_series` (уже отдаёт `received_at`).

### 7.2 Доверие — `network` как **ungated** источник

- `server/trust/domains.py::DOMAIN_SOURCES` — **`"network"` НЕ добавляем** в фазе 1. Значит источник не образует доменного гейта и **не трогает скоры здоровья дня-1**.
- `server/pipeline.py::evaluate_trust()` пройдёт по `source_health` и увидит ключ `"network"`: фиксирует `collector_status`/свежесть через `upsert_source_trust(...)`, но домена для него нет → нет гейтинга.
- Семантическая валидация — **только для решающих сигналов** (materiality governor). Сеть в фазе 1 не решающая → регистрируем `network` как collector-only источник **без семантического валидатора** (no-op в `validate_source`).
- **Проверка при реализации (open question):** убедиться, что `evaluate_trust`/`_extract_reading`/`validate_source` корректно обрабатывают неизвестный источник `network` без ошибки (ожидаемо да; покрыть тестом).

### 7.3 Минимальный показ (видимое изменение фазы 1)

- Блок на `device.html` «Сеть (сбор)»: последние карты (имя/IP/шлюз/статус), число соседей, число связей, статус источника `network` (ok/partial/blocked). Read-only.
- Полноценная **карта — фаза 2**. Здесь — только подтверждение, что данные доходят (для smoke + оператора).

## 8. Фазы 2 и 3 (контекст-ориентир, НЕ реализуются здесь)

**Фаза 2 — карта + аналитика (сервер, своя спека):**
- **Карта:** union соседей со всех агентов; сопоставление MAC/IP соседа с собственными картами известных агентов → «этот сосед = машина X с нашим агентом»; offline OUI-таблица → вендор неизвестного устройства; граф по подсетям/шлюзам. Дашборд — vanilla-JS (правило проекта: без SPA/websocket); конкретная библиотека рендера графа — решение спеки фазы 2.
- **Ось риска `network_health`:** новый `server/analytics/network_risk.py::compute_network_risk() -> Score100` (паттерн `storage_risk`/`battery_risk`); тренды по `network_quality` через `server/analytics/trends.py` (Theil-Sen + ETA); проводка в `recompute_scores` + `/api/v1/diagnostics` + блок `device.html`. Тогда же `"network"` добавляется в `DOMAIN_SOURCES`.
- **Аномалии флота:** «вся подсеть теряет пакеты → проблема инфраструктуры, не ПК» — по паттерну `server/analytics/fleet_anomaly.py` (когорта по подсети/`site_code`). Снимает массовые ложные «железные» тревоги.

**Фаза 3 — активный скан (за гейтом):**
- ping-перебор подсети + проверка портов; найдёт устройства без агента.
- **Выключен по умолчанию**, включается явной настройкой, rate-limit, только с письменного разрешения (выглядит как атака для EDR). Гейт: реальная потребность + согласование безопасности.

## 9. Тестирование (фаза 1)

Расширяем существующие наборы (`tests/test_collectors_parsers.py`, `tests/test_locale_fixtures.py`):
- **Golden-фикстуры коллектора:** записанный JSON-вывод PowerShell → парсер. Проверяем разбор карт/соседей/связей/качества, приведение чисел, квирк `as_list`.
- **Локаль-фикстуры:** вывод русской Windows → парсер не ломается (язык-независимость фактом; enum'ы английские на любой локали).
- **Фильтр приватности:** внешний `remote_ip` отброшен; внутренний — оставлен; адрес сервера — оставлен; loopback — отброшен.
- **Частичный сбой:** один под-вызов `blocked` → блок пуст + статус `partial`, остальные блоки есть.
- **Статусы:** `ok`/`partial`/`empty`/`blocked` маппятся верно; нет «тихого None».
- **Контракт:** старый конверт без сетевых полей валиден; новые поля round-trip; версия не меняется (forward/back-compat).
- **Пайплайн+доверие:** ingest `historical` с сетью → данные в payload; `evaluate_trust` фиксирует источник `network` ungated; скоры здоровья дня-1 **не изменились**; кейс «нет сети» → источник `empty`, скоры не тронуты.
- **Типизация:** `mypy` на `client/` чистый.

**Гейты до «готово»/merge:** `make check` (ruff · mypy[server+shared+client] · bandit · pytest cov ≥80%) зелёный · `python smoke.py` OK · строка в `CHANGELOG.md` в том же коммите.

## 10. Открытые вопросы / проверки при реализации

1. **Неизвестный источник в доверии:** подтвердить, что `evaluate_trust` не падает на `network` без домена (покрыть тестом). Если требуется регистрация — добавить `network` как collector-only без семантического валидатора.
2. **Механизм ping:** `Test-Connection` (ICMP, может быть закрыт фаерволом) для шлюза/DNS vs stdlib TCP-connect к серверу. План: ICMP где можно, TCP-connect к серверу как опора; блокировано → `partial`, не ошибка.
3. **Объём:** финальные капы списков соседей/связей (предв. 256) — подобрать по реальному снимку; защита строки `historical.payload` от раздувания.
4. **Каденс historical:** подтвердить текущую периодичность сбора `historical` в цикле агента и переиспользовать (не вводить новый таймер).
5. **Гранулярность source_health:** один ключ `network` (фаза 1) против под-ключей (`network.adapters`/`.map`/`.quality`). Берём один; под-ключи — возможное уточнение позже.
6. **Wi-Fi сигнал/канал — язык-независимый источник:** найти числовой CIM-класс (напр. `root/wmi` `MSNdis_80211_*`) для сигнала/канала; если на целевых Windows он недоступен/ненадёжен — `signal_pct`/`channel` **не собираем** (оставляем SSID + факт Wi-Fi). `netsh wlan` запрещён (локализован).
