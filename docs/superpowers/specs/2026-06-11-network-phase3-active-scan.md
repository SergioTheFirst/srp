# Сеть — фаза 3: активный поиск устройств (скан за гейтом)

**Дата:** 2026-06-11
**Статус:** дизайн (производный от `2026-06-10-network-tools-design.md` §8, фаза 3)
**Объём:** агент + контракт (аддитивно) + минимальный показ на сервере.

---

## 1. Цель

Фазы 1–2 видят только то, что машины уже знают (ARP). Устройство, с которым давно
не общались, в ARP отсутствует → его нет на карте. Фаза 3 закрывает дыру:
**ping-перебор СВОЕЙ /24-подсети** наполняет ARP-таблицу, существующий конвейер
(Ф1 соседи → Ф2 карта) автоматически показывает найденное. Плюс **проверка
фиксированного набора портов** у ответивших — подсказка типа устройства
(9100 = принтер, 445 = Windows, 3389 = RDP, 80/443 = веб-морда).

## 2. Гейт безопасности (главное решение)

| # | Решение | Почему |
|---|---------|--------|
| G1 | **Выключен по умолчанию**: `ClientConfig.network_scan_enabled = False`; включение = явная правка config.json на конкретной машине | Активный скан выглядит как атака для EDR/IDS; включать можно только там, где есть письменное разрешение владельца сети — это ответственность оператора, агент лишь даёт возможность |
| G2 | **Цели и порты НЕ конфигурируются**: только /24 вокруг собственного RFC1918-адреса; порты — фиксированный список в коде (80, 443, 445, 3389, 9100) | Конфигурируемые цели/порты превращ
